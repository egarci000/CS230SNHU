# CS230 SNHU Software Design Document



&nbsp;	For this course, I worked on a Software Design Document that showcased my skills identifying how a software application should be designed when needing to expand to many different operating systems. I designed a plan to take The Gaming Room's existing game and scale it to many platforms. I also advised the client company on which operating system to host their server on and identified the features that operating system provides when hosting a game application on it.





&nbsp;	The Gaming Room wanted us to take their existing mobile game called Draw It Or Lose It and design it so it could scale to many operating system platforms. Throughout the documentation, I believe I excelled in crafting a cohesive plan to design this application so it meets all their needs. I used ample research to craft this design document and took steps to make it as clear as possible so that someone who didn't know much about software design, could look at and understand. 





&nbsp;	When developing the code for the application, one thing I found helpful from working through the design document was writing down the design constraints. This helped me develop the code because, from the beginning, I knew the requirements of building such an app, as well as the limitations. From there, I was able to make further decisions on how to craft the classes and how to make them connect to each other. I was also able to clearly articulate to the client and to any reader of the design document the features that were going to be added to the platform to scale it from just a mobile platform to supporting many different operating systems, such as Windows and MacOS.



&nbsp;	

&nbsp;	If I could pick one section of my work to revise on the Software Design Document, I would revise the Domain Model section. This is where I take the UML diagram of the main class structure of the application and expand on the role of each class and how they connect. I would add more detail and present a better holistic view of the interconnected classes.



&nbsp;	

&nbsp;	When designing an application and the design document, it is always important to consider the user's needs. We want to fully know what their idea for the application is, as well as any concerns that may arise during development. This is important because, from listening well to the user we can know what features they want built, their timeline, and other relevant factors that support development. During the work on designing the software, I was able interpret and consider the user's needs by reading through their requirements, crafting an understanding, and adding that to my design document. From there, I was able to start work on designing the game software's classes and identifying where to host the server application. 



&nbsp;	I approached designing software by first taking a UML diagram and writing down its different classes and how they connect. Then, I designed a list of features the server application would need in order to be hosted on many different operating platforms. Finally, I researched three operating platforms the application could realistically live on: Windows, MacOS, and Linux. I then identified Linux as my choice, given its low licensing fees and ample documentation and support. In the future, I will use the same structure in designing a similar software application. I will also make sure to always keep the client involved as needed and work to give a clear, holistic view on the design of an application.
